# Codex 红绿灯多会话自适应模式实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在保持单个 Codex CLI 时现有小红绿灯体验不变的前提下，检测到多个 Codex CLI 会话时自动提供“极简悬浮任务抽屉”，让用户能看清每个任务窗口的红/黄/绿状态。

**Architecture:** 现有 `codex_traffic_light_state.json` 是单状态文件，多个 CLI 会互相覆盖；本计划升级为“每会话一个状态文件”的多会话状态目录，WPF 聚合这些文件并按优先级显示。UI 使用自适应模式：1 个活跃会话时只显示现有红绿灯，2 个及以上时显示数量角标，用户点击齿轮或出现黄灯时展开任务抽屉。

**Tech Stack:** .NET 8、C#、WPF、xUnit、PowerShell hook script、`FileSystemWatcher`、`System.Text.Json`。

---

## 结论与把握程度

我不能说“百分之百有把握”，原因是：目前尚未实测 Codex hook 进程在 Windows 下能否稳定提供可区分每个 CLI 的会话身份。这个风险不能靠想象消除，必须先做探针验证。

经过漏洞审计后，本计划把不确定性收敛为一个前置任务：先记录 hook 运行时的环境变量、父进程链、工作目录和进程 ID。若能稳定找到触发 hook 的 Codex CLI 进程，则采用“父进程 PID + 进程启动时间”作为会话 ID；若不能，则启用备用的 `codex-light.ps1` 启动器方案，通过环境变量显式注入会话 ID。这样即使首选识别方案失败，功能仍有可靠落地路径。

## 用户体验规则

1. 单任务模式：
   - 只有 1 个活跃 Codex CLI 会话时，窗口保持当前小红绿灯外观。
   - 不显示任务抽屉，不改变默认占屏面积。

2. 多任务模式：
   - 发现 2 个及以上活跃会话时，齿轮按钮旁显示数字角标，例如 `2`、`3`。
   - 默认仍不强制展开，避免突然变宽。
   - 点击齿轮时展开/收起右侧任务抽屉。

3. 黄灯提醒：
   - 任意会话从非黄灯变为黄灯时，抽屉自动展开 2-3 秒。
   - 若用户手动收起后 10 秒内再次出现黄灯，不重复弹开，避免打扰。

4. 聚合主灯优先级：
   - 任意会话为黄灯：主灯显示黄。
   - 否则任意会话为红灯：主灯显示红。
   - 否则全部为绿灯：主灯显示绿。
   - 无活跃会话或状态损坏：主灯显示灰/暗。

5. 抽屉排序和显示：
   - 排序：黄灯 > 红灯 > 最近更新的绿灯。
   - 每行显示：状态点、项目名、路径尾部、持续时间、状态标签。
   - 绿色完成项降低透明度，不抢注意力。
   - 超过 5 个会话时抽屉内部滚动，不继续扩宽。

## 漏洞审计

### 第一轮审计：会话身份是否可靠

**漏洞：** hook 命令可能没有官方稳定的 session id；多个 Codex CLI 写状态时，如果无法区分来源，就无法显示“每个窗口”的状态。

**解决方案：**

1. 增加前置探针任务，不直接改 UI。
2. hook 脚本先输出诊断文件：环境变量、当前进程 ID、父进程 ID、父进程链、当前目录、事件名。
3. 若父进程链能找到不同 Codex CLI 进程，则使用 `processId + processStartTime` 生成会话 ID。
4. 若父进程链不可靠，则提供 `codex-light.ps1` 启动器，启动 Codex 前设置：
   - `CODEX_TRAFFIC_LIGHT_SESSION_ID`
   - `CODEX_TRAFFIC_LIGHT_SESSION_NAME`
   - `CODEX_TRAFFIC_LIGHT_WORKING_DIRECTORY`

**审计结果：** 仍有不确定性，但有明确验证任务和备用路径，不会阻塞整体方案。

### 第二轮审计：多个 CLI 同时写文件是否会损坏状态

**漏洞：** 如果多个 hook 同时写同一个 `sessions.json`，会产生竞争：后写覆盖先写，甚至写出半截 JSON。

**解决方案：**

1. 不使用单个 `sessions.json` 作为首选存储。
2. 每个会话写自己的状态文件：

```text
%CODEX_HOME%\codex-traffic-light\sessions\<sessionId>.json
```

3. hook 脚本写入时采用临时文件 + 原子替换：

```text
<sessionId>.json.tmp -> <sessionId>.json
```

4. WPF 监听整个 `sessions` 目录，读取所有 `*.json` 后聚合。

**审计结果：** 这个方案消除了多会话写同一文件的主要竞争风险。

### 第三轮审计：PID 复用和陈旧会话

**漏洞：** Windows 进程 ID 可能复用；已关闭的 Codex CLI 会话可能长期留在列表里。

**解决方案：**

1. 会话 ID 不只用 PID，而用 `PID + StartTimeTicks`。
2. 状态文件包含：
   - `processId`
   - `processStartTime`
   - `updatedAt`
   - `workingDirectory`
3. WPF 聚合时检查：
   - 进程不存在：标记为完成/离线。
   - 绿灯超过 5 分钟：默认隐藏，可在托盘菜单“显示已结束会话”里查看。
   - 黄灯超过 5 分钟无更新：隐藏，避免继续误导为等待操作。
   - 红灯超过 10 分钟无更新：隐藏，避免旧处理状态长期占位。

**审计结果：** PID 复用和陈旧会话都有可检测、可清理的规则。

### 第四轮审计：自动展开会不会打扰

**漏洞：** 多任务时抽屉自动展开可能遮挡屏幕，反而让小工具变烦。

**解决方案：**

1. 只有黄灯状态才自动展开。
2. 自动展开持续 2-3 秒后自动收起。
3. 用户手动收起后设置 10 秒冷却。
4. 托盘菜单提供开关：`黄灯时自动展开任务抽屉`。

**审计结果：** 自动展开被限制在真正需要用户动作的场景，可关闭。

### 第五轮审计：单任务体验是否会被破坏

**漏洞：** 新功能可能让用户单任务时也看到额外 UI，破坏当前满意的小红绿灯。

**解决方案：**

1. 活跃会话数为 0 或 1 时，保持当前窗口宽高和布局。
2. 数量角标只在活跃会话数大于 1 时显示。
3. 抽屉仅在多会话模式下可展开。
4. 现有单状态文件 `codex_traffic_light_state.json` 保留一段兼容期，避免新逻辑失败时完全没状态。

**审计结果：** 单任务体验可以保持不变。

## 最终风险结论

这套方案的最大漏洞是“会话身份是否能自动识别”。计划已经把它提前为任务 0，并准备了启动器兜底。第二大漏洞是“多进程并发写同一文件”，计划改为每会话独立文件，避免主要竞争。第三大漏洞是“UI 变乱”，计划采用单任务不变、多任务角标、按需抽屉、黄灯短暂自动展开的克制交互。

因此，在执行任务 0 之前，不能宣称百分之百确定自动识别可行；但在任务 0 验证通过，或切换到启动器方案后，整体设计有明确实现路径和可验证验收标准。

## 2026-06-01 追加：VS Code 插件会话修复

已确认根因：

- 旧实现只靠 Codex 进程 PID 和工作目录生成会话 ID。
- VS Code 插件通过共享的 `codex.exe app-server` 触发 hook，多个聊天会落到同一个 PID 和同一个工作目录。
- 旧实现还会按工作目录合并会话，所以两个插件聊天会被压成一个。
- 任务名旧逻辑只取工作目录末尾，因此会显示 `Mes`、`28022`、`System32` 这类不直观名称。

修复策略：

- hook 脚本读取 Codex 传入的 stdin JSON，优先使用 `session_id` 作为稳定会话 ID。
- `UserPromptSubmit` 事件读取 `prompt`，截取为任务显示名；后续 `Stop` 等无 prompt 事件复用旧显示名。
- 工作目录优先使用 stdin 的 `cwd`，其次才是环境变量或当前目录。
- 只有弱身份会话（如 `pid-*-unknownstart`）才按工作目录去重；可靠 `session_id` 不再按目录合并。

- 目录 watcher 增加 5 秒刷新，用于在进程关闭但没有新 hook 事件时自动更新可见列表。

仍需用户侧注意：

- 更新后的 EXE 第一次启动会重写 hook 脚本。
- Codex/VS Code 里如果提示 hooks 信任，需要重新信任。
- 对 VS Code 插件，“聊天关闭”目前没有独立 hook；当前以短时隐藏规则兜底，再次在旧聊天发送消息会立即用同一个 `session_id` 恢复监控。
